# 前端 DEMO 开发交接约束

更新日期：2026-06-08

这份文档用于你在我暂停期间，把前端 DEMO 页面设计与开发工作交给其他工具或其他对话继续推进。

目标只有一个：

- 允许外部工具继续把页面体验做深做细；
- 同时保证这些结果后续能平滑接回当前仓库，而不是变成一套漂亮但难以并入的孤岛页面。

---

## 1. 项目背景

当前项目不是纯展示型站点，而是一个 **AI 服务设计工具箱**。

现阶段仓库里已经存在的正式基础包括：

- 应用壳：`src/app/*`
- 设计系统：`src/design-system/*`
- 工具运行时：`src/domain/*`、`src/application/*`
- 第一个正式工具：`src/tools/journey-map/*`
- AI 对话区：`src/features/assistant/*`
- 账号页：`src/features/account/*`
- CloudBase 文档持久化链路

因此，新的前端 DEMO 设计与开发，不能再按“一次性静态 landing page”思路做，而要按 **可回接到现有模块化工程** 的方式做。

## 3. 设计目标

外部工具继续做 DEMO 时，优先围绕下面 4 类页面推进：

1. 工具工作台页
2. 工具列表 / 工具入口页
3. 项目视图页
4. 账号 / 订阅 / 用量相关页面

请注意，这里的“做页面”不是要求把所有后端功能都接通，而是要求：

- 信息架构清楚
- 组件边界清楚
- 视觉语言统一
- 后续能把真实数据与真实行为接上去

---

## 4. 必须遵守的页面设计原则

### 4.1 工具优先，不做营销页思路

这个产品的核心是“用户进入后尽快开始使用工具”，不是让用户看大段品牌宣言。

因此页面风格应偏：

- 专业工作台
- 低噪声
- 内容优先
- 高信息密度但不混乱

不要偏成：

- SaaS 营销首页
- 资讯平台
- 社区产品
- 作品集网站

### 4.2 AI 是协作层，不是页面装饰

AI 在本产品里不是一个悬浮彩蛋，而是与工具并列的主交互区域。

页面上要能自然容纳：

- 当前工具上下文
- AI 澄清问题
- AI proposal 预览
- 用户确认后应用
- 后续跨工具推荐

因此不允许把 AI 区域做成“随便塞个聊天框”。

### 4.3 项目上下文必须可见

当前仓库已经进入正式 `projectId / documentId / revision` 体系。

所以后续新 DEMO 页面在视觉和结构上，要给这些概念留下位置：

- 当前项目
- 当前工具文档
- 最近修改
- 保存状态
- 版本 / revision

即使先做静态展示，也要有这些信息位。

### 4.4 多工具扩展必须提前考虑

Journey Map 只是第一个工具。

因此前端设计不能把全局框架写死成只适配 Journey 的结构。需要预留：

- 工具导航
- 工具卡片
- 工具推荐
- 项目中多个工具产物并存
- 从一个工具跳到另一个工具

## 5. 技术与代码约束

### 5.1 必须兼容当前技术栈

后续要并回当前项目，所以默认技术边界是：

- React
- Vite
- TypeScript
- Chakra UI
- 现有 `src/app` / `src/design-system` / `src/features` / `src/tools` 模块结构

允许：

- 在 Chakra UI 基础上做适度定制
- 新增产品级组件
- 优化布局、页面层级和交互细节

不允许：

- 改成另一套主 UI 框架
- 引入大而重、与 Chakra 重叠的第二套组件体系
- 把核心页面写成脱离现有项目结构的独立原型工程

### 5.2 组件拆分必须可回接

新增页面或组件时，优先落到这些边界：

- `src/app/*`：应用壳、导航、页面切换、全局工作台结构
- `src/design-system/*`：通用视觉组件、字段、按钮、卡片、布局原语
- `src/features/*`：AI、账号、账单、项目视图等产品 feature
- `src/tools/*`：具体工具页面

不要：

- 再把大量实现塞回 `src/App.tsx`
- 建立一个完全平行、未来没人敢合并的新目录体系

### 5.3 业务逻辑与展示逻辑分离

如果只是做 DEMO 页面，可以做静态占位，但要明确标记哪些是：

- 纯视觉占位
- 假数据展示
- 后续要接真实接口
- 后续要接真实工具运行时

不要在展示组件里偷偷写死：

- AI 协议
- 文档保存规则
- revision 规则
- 支付扣次逻辑
- CloudBase 认证逻辑

### 5.4 必须使用独立设计分支

这次外部前端 DEMO 的默认工作方式不是直接在当前主开发分支上推进，而是：

- 先在本地 Git 拉一个独立的前端设计分支
- 在这个分支上进行页面探索、样式调整和结构优化
- 暂时不假定整条分支都会并入主项目

建议分支名：

- `codex/frontend-demo-design`
- `feat/frontend-demo-exploration`
- `design/frontend-workbench-refresh`

后续等我恢复后，再重新判断：

- 哪些提交可以吸收到主项目
- 哪些只保留设计思路
- 哪些需要按当前模块边界重写后再并入

## 6. 优先开发顺序

如果时间有限，建议外部工具按这个顺序推进：

### 第一优先级：整体工作台体验

先统一这些部分：

- 顶部全局导航
- 左右分栏或主辅区域关系
- 工具页面通用壳层
- 项目信息位
- 保存状态位
- AI 协作区视觉结构

### 第二优先级：工具入口与项目页

建议补强：

- 工具首页 / 工具选择页
- 项目列表页
- 单个项目的产物概览页
- “从项目进入工具”的路径

### 第三优先级：账号与订阅页

建议补强：

- 登录后账户页
- 次数余额 / 用量页
- 订单 / 套餐状态页

这些页面先做结构和体验，不要求先接通真实支付。

### 第四优先级：第二个工具的静态骨架

如果还有余力，可以先做第二个工具的页面骨架，例如：

- Personas
- Stakeholder Map
- Service Blueprint

但只做结构，不要另起一套运行时规则。

## 7. 页面产出要求

如果对方是做设计稿或前端 DEMO，请要求对方最终至少交付这些内容：

### 7.1 页面清单

明确做了哪些页面，例如：

- 工具工作台页
- 工具入口页
- 项目页
- 账号页
- 订阅页

### 7.2 组件清单

明确新增了哪些组件，例如：

- AppShell
- ProjectSwitcher
- ToolCard
- UsageSummaryCard
- BillingPlanCard
- AssistantSidebar

### 7.3 组件归属建议

每个新增组件都要标清楚后续应该归到：

- `app`
- `design-system`
- `features`
- `tools`

### 7.4 静态占位说明

必须明确哪些地方现在只是静态占位，例如：

- 假数据项目列表
- 假余额数字
- 未接通的支付按钮
- 未接通的工具推荐结果

这一步很重要，否则后面回接时会很难判断哪些代码能直接留，哪些只是演示层。

### 7.5 分支与提交说明

必须额外回传：

- 当前独立分支名
- 这一轮前端设计探索包含哪些提交
- 哪些提交是视觉探索
- 哪些提交已经具备较强回接价值

## 8. 禁止事项

外部工具在开发时，明确禁止这些行为：

1. 禁止改坏当前 Journey 工具的正式运行链路
2. 禁止把静态 DEMO 直接覆盖现有正式工具页面
3. 禁止新建一套脱离仓库结构的独立 demo app
4. 禁止把支付、认证、AI、持久化逻辑伪装成“已经完成”
5. 禁止大量引入后续难以维护的动画、装饰和纯视觉组件
6. 禁止为了好看牺牲工具型产品的信息效率
7. 禁止无边界地改全局样式
8. 禁止把前端设计探索分支直接视为主项目正式实现分支

---

## 9. 最理想的交付方式

如果你把任务交给其他工具，最理想的结果不是“对方做出一套完整新产品”，而是：

- 在当前产品方向内，给出更成熟的前端页面方案
- 尽量直接在现有项目结构里落代码
- 明确哪些是演示层，哪些是真正可回接资产

最适合后续接回当前仓库的交付物包括：

1. 可直接放进现有仓库的页面代码
2. 新增组件清单
3. 页面结构图
4. 静态占位说明
5. 设计说明与截图
6. 独立分支名与提交说明

## 11. 后续回接时我会怎么接

等我恢复后，我会按这个顺序接你这段时间的外部成果：

1. 先审信息架构是否符合当前产品方向
2. 再审这个前端设计独立分支里哪些提交只是探索，哪些值得吸收
3. 再审组件是否能并回现有模块结构
4. 再决定哪些页面代码直接吸收，哪些只吸收视觉与布局方案
5. 最后把真实业务逻辑接回这些页面

所以你这段时间让其他工具继续推进时，**最重要的不是“做得多”，而是“做得能回接”**。
