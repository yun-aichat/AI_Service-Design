# 服务设计 AI 工作流产品与商业模式草案

更新日期：2026-06-22

## 1. 当前产品定义

当前产品不应定义为笼统的“服务设计工具包”，而应定义为：

> 一个面向服务设计从业者与学生的自助式 AI 工作流工具。
> 用户先沉淀 Persona 资产，再通过自然语言输入场景，由 AI 自动选取相关 Persona，生成 Journey 初稿并给出关键落差判断。

这个定义强调的是一条连续工作流，而不是一组分散工具。

## 2. 当前阶段的产品边界

### 2.1 当前可售卖的产品

当前阶段的可售卖产品是：

- `Persona -> Journey` 的连续工作流
- Persona 作为上游输入资产
- Journey 作为第一个核心收费产出

### 2.2 当前不应混入产品定义的内容

以下内容属于平台愿景或后续路线，不应与当前商业模式混为一谈：

- 利益相关者地图
- 价值主张
- 更多下游服务设计工具
- 完整“工具包”叙事

这些内容可以保留在路线图中，但不应代替当前产品定义。

## 3. 核心用户与使用方式

当前目标用户包括：

- 服务设计咨询公司从业者
- 在校设计专业学生
- 企业内部从事服务体验、流程设计、用户研究相关工作的员工

产品当前是工作用途产品，但直接面向个人用户售卖。

当前预期的第一阶段获客方式：

- 先在用户既有学校圈层内推广
- 再向服务设计咨询公司中的熟人关系推荐
- 先在小范围内部使用中打磨，再逐步扩大推广

## 4. 核心用户价值

用户真正购买的，不是“做 Persona”和“做 Journey”这两个动作本身，而是：

> 更快得到一份能继续讨论、汇报、修改的服务分析初稿。

当前产品提供的核心价值包括：

- 减少整理调研资料的时间成本
- 降低从空白页开始组织信息的难度
- 基于 Persona 资产快速进入 Journey 推演
- 帮助用户快速得到一个可编辑、可继续完善的初版结果

因此，产品卖点不应只描述为“有这些工具”，而应描述为：

> 把零散调研和场景想法，快速整理成可编辑 Persona 与 Journey 初稿的 AI 工作台。

## 5. 当前工作流定义

当前阶段的核心工作流如下：

1. 用户先在 Persona 工具中沉淀用户画像资产
2. 用户在 Journey 场景中通过聊天描述服务场景
3. AI 从 Persona 库中选取相关核心用户
4. AI 基于 Persona 与场景生成 Journey 初稿
5. AI 输出关键落差、风险、机会点等判断
6. 用户在结果上继续手动编辑、修正和导出

这个工作流的重点不是“两个工具都存在”，而是：

> 上游 Persona 资产必须能够直接提升下游 Journey 推演质量。

如果 Persona 只是文本堆积，Journey 仍需要 AI 从头重新理解，则工作流不成立，只是页面跳转。

## 6. 当前商业模式定义

### 6.1 商业模式一句话

当前商业模式可以定义为：

> 免费提供基础整理与手动编辑，按次收费出售 AI 结果生成。

### 6.2 免费边界

当前适合免费或低门槛提供的能力：

- Persona 基础整理
- Persona 资产沉淀
- 已生成结果的手动编辑
- 已有文档的查看、修改与导出

补充规则：

- Persona 初稿生成仍属于免费或低门槛入口能力
- 但免费入口不等于无限 AI 使用
- Persona 导入生成应受模型、次数或资料规模边界约束
- 当前首个明确收费点仍然是 Journey 推演结果生成

### 6.3 付费边界

当前第一个明确收费点应定义为：

> 基于 Persona + 场景，生成一版 Journey 初稿及关键落差判断。

用户付费买到的是一个明确结果，而不是抽象的“AI 使用权”。

## 7. 收费单位建议

当前阶段最适合先钉死的收费单位是：

> 一次完整的 Journey 推演结果

这个结果至少应包含：

- 本次采用的 Persona 视角
- Journey 初稿
- 关键落差判断
- 风险、机会点或关键发现摘要

不建议一开始就向用户暴露以下计费逻辑：

- 按 token 计费
- 按聊天轮数计费
- 按编辑时长计费
- 按所有 AI 行为统一扣费

对外应按结果单位售卖，对内再用模型成本与 token 进行核算。

## 8. 当前最关键的产品与商业风险

### 8.1 最大风险

当前最大风险不是工具数量不够，而是：

> 收费链路的输出是否足够稳定，且不依赖创始人本人。

由于当前产品目标是“相对标准化的自助工具”，因此用户购买的是：

- 不需要创始人陪跑也能使用
- 输入基础信息后能稳定得到过得去的结果
- 结果虽然不是终稿，但能显著减少前期整理与推演工作

如果这点做不到，产品就会从“自助工具”退化为“被工具包装的专家服务”。

### 8.2 工作流成立条件

`Persona -> Journey` 连续工作流是否成立，关键取决于：

- Persona 字段是否足够结构化
- Persona 资产是否真的能复用
- Persona 资产是否能稳定提升 Journey 生成质量
- 同类输入是否能得到相对稳定的输出

如果这些规则不清晰，平台感只是叙事，不是产品事实。

## 9. 当前需要尽快钉死的产品规则

为了让收费模型可落地，当前应优先明确以下规则：

1. 用户输入什么信息，才允许触发一次 Journey 生成
2. AI 如何从 Persona 库中选取相关核心用户
3. 一次完整 Journey 推演结果的标准输出包含哪些部分
4. 什么情况下结果不完整、不应扣费
5. 用户补充信息后再次生成，是否算新的一次收费

这些规则决定：

- 首页如何销售
- 聊天交互如何设计
- Journey 工作台如何承接结果
- 计费逻辑如何避免“乱扣次”的感知

## 10. Journey MVP 扣费规则

MVP 阶段不建议一开始就区分过多收费挡位。当前更适合先验证：

> 用户是否能理解什么时候会扣费，并愿意为第一次完整的 Journey 推演结果付钱。

因此，MVP 阶段先采用单一核心收费动作：

> 每次成功生成一版新的 Journey 初稿及关键判断，扣费一次。

### 10.1 基础规则

1. 澄清阶段不收费
2. 首次完整生成收费
3. 手动编辑不收费
4. 小幅 AI 协助先不直接收费
5. 用户确认重新推演后，才算新一次收费
6. 新场景、新 Persona 或新增足以改变结论的关键信息，按新一次生成收费

### 10.2 引导式再推演机制

为了减少用户在当前结果上持续小改动带来的“无限白嫖”风险，MVP 可加入一条非强制的引导机制：

- 当系统检测到小改动累计达到阈值后，弹出提示
- 默认阈值可先设为 `5` 次小改动
- 提示文案建议为：

> 当前内容已发生较多变化，是否基于最新内容重新执行一次推演？

这条机制的原则是：

- 只做引导，不强制扣费
- 让用户理解“手动编辑”和“重新推演”是两件事
- 当用户明确选择重新推演时，才触发新一次收费

### 10.3 当前阶段不建议做的事情

MVP 阶段先不要急于拆分以下复杂收费结构：

- 初步生成 / 深度生成多档计费
- 按聊天轮数计费
- 按 token 向用户直接计费
- 按“所有 AI 输出”统一扣费

更合适的顺序是：

1. 先把首次完整 Journey 推演的收费规则讲清楚
2. 先观察用户是否接受当前扣费逻辑
3. 等出现明显的价值分层后，再考虑拆出多档位

## 11. Journey 生成入口 SOP 草案

当前 MVP 不建议采用“纯结构化表单先填写完再生成”的重流程，也不建议采用“无限开放式对话澄清后再生成”的重聊天流程。

更合适的入口形态是：

> 一句话自然语言输入 -> AI 提取候选结构 -> 用户确认 -> 扣费生成

这个方案的目标是同时满足：

- 启动足够快
- Persona 资产可复用
- 收费边界清晰
- 澄清成本可控

### 11.1 入口原则

Journey 入口应定义为“聊天驱动的结构化生成”，而不是“聊天本身就是产品”。

AI 的职责不是无限陪聊，而是：

1. 尽快把用户的一句话需求收口成可确认的结构化条件
2. 在必要时只追问最关键的缺失信息
3. 在信息足够时主动进入“准备生成”状态

### 11.2 最小生成条件

当以下信息已确认后，Journey 可以进入可生成状态：

- 已确认一个 Persona
- 已确认服务情境
- 已确认核心任务
- 已确认本次分析范围或体验片段

如果四项中有缺失，AI 应继续补齐缺失项，而不是直接生成。

### 11.3 推荐交互流程

1. 用户用一句话描述需求
2. AI 先从输入中提取候选 Persona、候选场景、核心任务与分析范围
3. 如果信息已基本足够，AI 直接展示确认卡片
4. 用户对确认卡片进行选择或补充
5. 系统提示“确认后将消耗积分生成 Journey 初稿”
6. 用户点击“确认并生成”
7. 系统执行积分预占、Journey 推演与结果返回
8. 结果成功返回后完成扣费

### 11.4 确认卡片建议结构

确认卡片建议至少包含以下字段：

- 使用的用户画像：候选 Persona 1 / 候选 Persona 2 / 其他
- 所在场景：候选场景 1 / 候选场景 2 / 其他
- 核心任务：已识别结果，可编辑
- 分析范围：例如预订前 / 到店 / 使用中 / 售后，可选或可编辑
- 其他补充：自由输入框

按钮建议：

- `确认并生成`
- `继续补充`

在 `确认并生成` 按钮附近，应明确提示：

> 确认后将消耗积分，生成 Journey 初稿与关键判断。

### 11.5 AI 对话行为约束

AI 在 Journey 入口阶段应遵循以下约束：

- 默认优先提取结构，而不是继续开放式闲聊
- 每次只追问一个最关键的缺失项
- 信息足够后必须主动进入确认卡片，而不是继续发散
- 不应在未确认 Persona 和场景时直接进入正式生成
- 不应在澄清阶段输出看似完整的 Journey 结果，避免模糊收费边界

### 11.6 防止澄清白嫖的处理方式

MVP 阶段不建议在澄清阶段直接收费，但需要限制无限消耗 token 的风险。

推荐做法：

- Journey 入口阶段优先使用低成本模型
- AI 优先把聊天收口成确认卡片
- 连续多轮仍未收口时，提示用户改用结构化补充
- 轮数阈值可作为后台参数存在，例如 `5` 轮

需要注意：

- `5` 轮不是产品定义，只是兜底参数
- 是否进入确认卡片，优先依据“信息是否足够”判断
- 只有在用户持续闲聊、反复补无效信息时，才触发轮数兜底

### 11.7 AI 在入口阶段的推荐话术

当信息基本足够时：

> 我已经整理出本次生成所需的核心信息。请确认使用的用户画像、场景与补充内容。确认后将消耗积分生成 Journey 初稿与关键判断。

当缺少关键信息时：

> 还差一个关键点我才能帮你生成更可靠的 Journey：这次主要分析的是哪个体验片段？

当对话轮数较多但仍未收口时：

> 为了更快生成结果，建议先确认以下结构化信息，我再继续为你生成。

### 11.8 产品判断

这套入口方案的本质不是“对话生成”，而是：

> 用自然语言降低启动门槛，再用结构化确认锁定收费边界。

因此，后续无论实现为聊天面板、侧边助手还是弹层确认，都应保持这个核心顺序不变。

## 12. Journey 消费 Persona 的执行模型

Journey 与 Persona 的关系不应只是“选中一个画像然后拼接进提示词”，而应定义为：

> 多 Persona 独立推演 -> 单份 Journey 汇总输出

### 12.1 执行原则

- 用户先确认生成条件层，而不是先手动画完整流程
- 生成条件层包括：场景、核心任务、分析范围、补充说明、选用 Persona
- 系统先基于生成条件生成默认流程骨架
- 每个 Persona 作为一个独立执行视角，沿同一份默认流程骨架参与推演
- Persona 之间不互相协商，不做多 agent 互相对话
- 每个 Persona 只基于自己的稳定资产和当前场景独立完成一次推演
- 系统最后再把多个 Persona 的结果做对比和汇总，产出一份 Journey 初稿

### 12.2 Persona run 主输入

Journey 在单个 Persona 视角下的推演，默认直接消费 Persona 的稳定层资产：

- 基础信息层
- 已确认 traits
- 需求 / 偏好 / 雷点
- 已纳入的行为洞察摘要
- 已纳入的场景洞察摘要
- 当前场景描述

默认不直接消费：

- 原始证据
- 用户原声全文
- 右侧池中未纳入的洞察

### 12.3 输入边界

当前阶段默认策略不是按洞察条数限制，而是按输入文字量限制。

字数阈值按**单个 Persona run** 计算：

- 软阈值：`3500` 字
- 硬阈值：`5000` 字

规则：

- `<= 3500`：完整 Persona 稳定层直喂
- `3500 - 5000`：进入压缩预警，可按策略压缩行为/场景洞察摘要
- `> 5000`：强制压缩行为/场景洞察摘要

压缩顺序：

1. 保留基础信息
2. 保留已确认 traits
3. 保留需求 / 偏好 / 雷点
4. 压缩行为洞察摘要
5. 压缩场景洞察摘要

原始证据始终不进入主输入。

### 12.4 正式运行协议建议

当前更适合固定 4 个正式对象：

- `JourneyGenerationRequest`
- `JourneySkeleton`
- `PersonaRunResult`
- `JourneySynthesisResult`

它们分别对应：

- 一次正式生成请求
- 默认流程骨架
- 单 Persona 视角推演结果
- 最终 Journey 汇总结果

### 12.5 单 Persona run 的最小输出

单个 Persona 视角下的推演结果，不应直接输出一整份独立 Journey 成品，而应输出可被汇总层消费的结构化结果。

最小输出建议包括：

- 本次使用的 Persona 标识
- 本次使用的场景描述
- 针对默认流程骨架中每个阶段 / 步骤的判断结果
- 用户想法
- 用户感受
- 用户行为
- 痛点
- 痒点
- 爽点
- 本 Persona 视角下的关键发现摘要

这里的作用不是让前端展示 `N` 份并列 Journey，而是为后续汇总层提供稳定的可合并输入。

### 12.6 Journey 最终交付的三层结构

Journey 最终输出应固定为三层，而不是“一个默认图再加一些备注”：

1. 默认流程层
2. Persona 推演汇总层
3. 汇总分析层

第一层默认流程层负责定义这次服务体验本身的骨架，包括：

- 阶段
- 步骤
- 触点

第二层 Persona 推演汇总层负责承接多个 Persona 独立行走流程后的合并结果，核心字段包括：

- 用户想法
- 用户感受
- 用户行为
- 痛点
- 痒点
- 爽点

第三层汇总分析层负责解释这些结果意味着什么，核心内容包括：

- 机会点
- 差异分析

如果后续要扩展风险、建议、优先级，也应挂在第三层，而不是污染前两层。

### 12.7 汇总输出原则

- 最终主交付是一份 Journey，而不是 N 份 Journey
- 默认流程层先定义骨架，再承接 Persona 汇总结果
- Persona 间差异需要保留，但应进入汇总展示和分析解释，不应拆成多份主成品
- 汇总层负责：
  - 提炼共性
  - 标出差异
  - 形成一份可编辑 Journey 初稿
- 汇总层不应脱离各 Persona 推演结果重新发明新故事

## 13. 当前版本的产品结论

当前可以先采用如下结论作为产品与商业模式母定义：

- 产品：服务设计 AI 工作流工具
- 核心结构：`Persona 资产 -> 场景输入 -> Journey 推演`
- 核心价值：把重的整理、分类、推演工作压缩成一次快速产出
- 目标用户：服务设计学生、咨询顾问与企业内部相关从业者
- 免费部分：Persona 沉淀、手动编辑、已有内容修改
- 首个收费点：基于 Persona + 场景生成 Journey 初稿，并给出关键落差判断
- 收费方式：按次购买 AI 结果生成，不按 token 直接向用户计价
- 护城河方向：资产可复用、推演更稳定、结果更能用，而不是单纯增加工具数量

一句话总结：

> 当前卖的不是“工具包”，也不是“单次聊天”，而是“基于 Persona 资产的一次 Journey 推演结果”。
